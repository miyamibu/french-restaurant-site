# システム設計レビュー用 事前整理

この文書は `bistro-reservation` リポジトリの実装を前提にしたレビュー準備メモです。コードと設定ファイルから確認できた内容を優先し、コードだけでは断定できない項目は `要確認` としています。

## 1. システム概要
- システム名: Bistro Joa 予約システム
- 目的: レストランの席予約受付、オンラインストア注文受付、店舗向け管理運用を1つの Web アプリで提供する
- 成功条件:
  - 予約が営業日・受付締切・席キャパシティの制約内でのみ成立する
  - 注文金額がサーバ側で再計算され、正しい支払いフローに乗る
  - 管理者が予約一覧、注文一覧、振込先情報を安全に確認・更新できる
- 失敗すると一番困ること: 過剰予約、重複注文、誤入金処理、誤配送につながる整合性崩壊

## 2. 利用者
- 利用者の種類:
  - 一般利用者: 予約申込、問い合わせ送信、オンラインストア注文
  - 管理者: 予約状況確認、予約ステータス変更、営業日設定、注文確認、入金確認、発送完了、振込先管理
  - 外部 AI エージェント: 予約は API 経由で完結可能、ストアはウォームハンドオフのみ
  - 定期実行ジョブ: 期限切れ注文キャンセル、履歴削除、予約リマインド
- 各利用者ができること:
  - 一般利用者は会員登録なしで予約と注文を送信する
  - 管理者は `/admin` と `/dashboard` 配下で運用操作を行う
  - AI エージェントは `/api/reservations` を使った予約完結、または `/booking?mode=agent` `/on-line-store/...?...mode=agent` への誘導を行う
- 管理者権限の範囲:
  - 予約一覧参照、個別予約参照、予約ステータス更新
  - 営業日休業設定の参照・更新
  - 注文一覧参照、注文アクション実行、銀行口座情報の参照・更新・削除
- 認証方式:
  - 管理画面/管理 API: Basic 認証
  - Cron API: `Authorization: Bearer $CRON_SECRET`
  - 一般利用者向け機能: 認証なし
  - エンドユーザアカウント機能: 現状なし

## 3. 主な業務フロー
- 主要ユースケース1: 席予約
  - 利用者が `/booking` で日付、時間、人数、コース、氏名、電話番号を入力
  - フロントは日次・月次空席 API で候補日を表示
  - `POST /api/reservations` でサーバ側バリデーション、締切確認、席キャパ判定、DB 登録、通知メール送信を実施
- 主要ユースケース2: オンラインストア注文
  - 利用者が商品をカートに入れ、顧客情報と支払い方法を入力
  - `POST /api/orders` で商品価格を再取得し、合計金額をサーバ側で再検証
  - 注文は `QUOTED` で作成され、その後支払い方法設定により `PENDING_PAYMENT` へ遷移する
- 主要ユースケース3: 管理運用
  - 管理者が予約を確認し、`CONFIRMED / CANCELLED / DONE / NOSHOW` を更新
  - 管理者が注文一覧を見て、入金確認、店頭受領確認、発送完了、キャンセルを実行
  - 管理者が銀行振込先情報を更新し、履歴を暗号化付きで保存する
- キャンセル/失敗時の流れ:
  - 予約は利用者自身の UI キャンセルは未実装で、電話導線で運用
  - 注文は期限切れで cron が自動キャンセルする
  - 注文 API や予約 API はバリデーションエラー時に機械可読コードを返す
- 二重送信や再試行の扱い:
  - 予約は PostgreSQL advisory lock + serializable transaction + リトライで競合制御
  - 注文と注文アクションは `Idempotency-Key` と `api_idempotency` テーブルで冪等化
  - 注文状態更新は `version` による楽観ロックで多重更新を防ぐ

## 4. データ
- 重要データ:
  - 予約: 日付、時間帯、人数、氏名、電話番号、要望、ステータス
  - 営業日: 日付、休業フラグ、備考
  - 注文: 顧客情報、配送先、商品明細、合計金額、支払い方法、支払い参照番号、状態、期限
  - 銀行口座: 銀行名、支店名、口座種別、口座番号、口座名義
  - 監査/補助データ: `order_actions`, `human_tokens`, `api_idempotency`, `bank_account_history`
- 個人情報の有無: あり
  - 氏名、電話番号、メールアドレス、住所、問い合わせ本文
- 金銭データの有無: あり
  - 注文合計、入金参照番号、振込先口座情報、入金確認結果
- 整合性が重要なデータ:
  - 予約キャパ計算に使う `Reservation.status`, `date`, `servicePeriod`, `partySize`
  - 注文の `status`, `payment_method`, `payment_reference`, `total`, `version`
  - 銀行口座情報とその履歴

## 5. システム構成
- フロントエンド: Next.js App Router + React + TypeScript
- バックエンド: Next.js Route Handler ベースの API
- DB:
  - 予約系: Prisma 経由の PostgreSQL
  - 注文/振込先/監査系: Supabase Postgres
- 外部連携:
  - Supabase
  - Resend / SendGrid
  - Vercel Cron
  - LINE 連携用環境変数はあるが、リマインド送信本体は未実装
- 非同期処理:
  - 予約メール送信
  - 注文確認メール送信
  - cron による期限切れ注文キャンセル
  - cron による古い履歴削除
  - cron による翌日予約リマインド準備
- 通知:
  - 予約受付メール
  - 注文確認メール
  - 問い合わせメール
  - LINE 通知は将来実装前提

## 6. リスク箇所
- 特に壊れやすい部分:
  - 予約キャパ判定ロジックと席パターン計算
  - 予約 API の同時実行制御
  - 注文状態遷移の RPC 群とアプリ側バリデーションの整合
  - 2系統 DB 運用による運用複雑化
- 外部依存:
  - Supabase 停止時は注文系と振込先管理が停止
  - PostgreSQL/Prisma 停止時は予約系が停止
  - メールプロバイダ停止時は通知配信が欠落
  - Vercel Cron 停止時は期限切れ注文が残留
- 人手運用が入る部分:
  - 予約キャンセル受付
  - 銀行振込の着金確認
  - 店頭支払い受領確認
  - 発送完了更新
  - 銀行口座情報更新
- 障害時に止まる部分:
  - DB 接続断で予約・注文とも停止
  - `CRON_SECRET` 不整合で cron が止まる
  - Basic 認証情報不整合で管理運用が止まる

## 8. 不変条件
- 絶対に壊れてはいけない業務ルール:
  - 休業日、当日、受付締切後、予約可能期間外の予約は成立してはいけない
  - 予約キャパシティを超える組み合わせは Web 完結してはいけない
  - 注文合計はクライアント値ではなくサーバ側商品価格で確定する
  - `payment_reference` は有効注文間で一意でなければならない
- 二重処理してはいけないもの:
  - 同一注文の作成
  - 同一注文への支払い方法設定
  - 同一注文への入金確認、店頭受領確認、発送完了
  - 同時刻帯の過剰予約登録
- 参照・更新してはいけない境界:
  - `SUPABASE_SERVICE_ROLE_KEY` をクライアントへ露出してはいけない
  - 管理系 API は Basic 認証なしで参照・更新してはいけない
  - AI ハンドオフ URL に氏名、電話、メール、住所などの個人情報を含めてはいけない

## 9. 状態遷移
- 主な状態一覧:
  - 予約: `CONFIRMED`, `CANCELLED`, `DONE`, `NOSHOW`
  - 注文: `QUOTED`, `PENDING_PAYMENT`, `PAID`, `SHIPPED`, `CANCELLED`
- 状態遷移図または遷移ルール:
  - 予約:
    - 新規作成時は `CONFIRMED`
    - 管理者が `PATCH /api/admin/reservations/[id]` で状態更新
  - 注文:
    - 作成時 `QUOTED`
    - 支払い方法確定で `PENDING_PAYMENT`
    - 入金確認または店頭受領で `PAID`
    - 発送完了で `SHIPPED`
    - `QUOTED` または `PENDING_PAYMENT` から `CANCELLED`
    - human confirmation は注文状態を変えず、支払い方法確定の前提条件を更新する
- 禁止遷移:
  - `QUOTED -> SHIPPED`
  - `SHIPPED -> PAID`
  - `SHIPPED -> CANCELLED`
  - `CANCELLED -> *`
  - `PENDING_PAYMENT -> QUOTED`
- 手動変更可能な状態:
  - 予約状態は管理画面から手動変更可能
  - 注文状態は管理画面のアクション経由でのみ変更可能
  - 注文一覧 API の直接 `PUT/DELETE` は無効化済み

## 10. 復旧と運用
- 失敗時の復旧方法:
  - `requestId` を起点にアプリログ、Vercel Logs、Supabase Logs を確認する
  - DB 障害時はバックアップから復旧し、必要なら Prisma migration / Supabase SQL の適用状態を見直す
  - デプロイ不良時は Vercel の前回成功デプロイへロールバックする
- 再実行方法:
  - 注文作成と注文アクションは同じ `Idempotency-Key` なら再送しても再実行されず再生応答になる
  - cron は `POST` または互換 `GET` で手動再実行可能
  - 予約作成はサーバ側で競合再試行するが、クライアント側冪等化キーは未実装
- 手動補正が必要なケース:
  - 予約キャンセル/無断キャンセル/来店完了の反映
  - 振込額不一致時の照合
  - 店頭支払い済みの確認
  - 銀行口座変更
  - メール送信失敗後の個別連絡
- 監査ログが必要な操作:
  - 注文状態更新
  - 入金確認失敗
  - 銀行口座更新・削除
  - 予約状態変更
  - 現状、注文系は `order_actions` がある一方、予約状態変更は専用監査テーブルがなくログ中心
- アラート/監視の有無:
  - JSON 構造化ログは実装済み
  - Vercel / Supabase のログ確認手順はある
  - 明示的な外形監視、しきい値アラート、通知先連携はコード上は未確認
